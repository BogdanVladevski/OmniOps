00

The whole engine in one loop

Every telemetry packet enters through Kafka, is processed exactly once (when possible), persisted with its domain events in one transaction, cached for fast reads, analysed for distress signals, and pushed live to mobile clients. Nothing is invented — the message is the source of truth; OmniOps deduplicates it, hardens it, and adds derived intelligence.

01

Ingesting from Kafka

Telemetry arrives on topic fleet-telemetry. The consumer runs as a BackgroundService with resilient reconnect logic — if the broker drops, it backs off and resubscribes rather than crashing the host. 

// KafkaTelemetryConsumer — continuous poll loop
_kafkaConsumer.Subscribe("fleet-telemetry");
var result = _kafkaConsumer.Consume(stoppingToken);
// → deserialize → KafkaTelemetryMessageProcessor.ProcessAsync(...)

The feed's non-obvious realities

Most of the engineering effort lives in failure routing, because a streaming pipeline will silently lose or duplicate data if errors are handled naïvely:

Scenario Why it bites How it's handled
Malformed JSON or missing fields Throwing aborts the consumer loop; the offset may never advance cleanly Parse failure → route raw payload to fleet-telemetry-dlq with reason
FluentValidation failure (empty vehicle ID, out-of-range values) Retrying won't fix bad data — you'd waste DB connections forever Validation exception → immediate DLQ, no retry
Transient Postgres failure mid-SaveChanges Dedup lock still held → packet stuck; or outbox rows orphaned in EF tracker Release lock, throw TransientProcessingException, retry with fresh DI scope
Retries exhausted Packet would loop forever or be silently dropped After ProcessingRetryMaxAttempts → DLQ with exhaustion reason
Unhandled consumer exception Message consumed but never processed — silent data loss Generic catch publishes to DLQ before continuing the poll loop

02

Processing — CQRS and idempotency

The handler is the heart of the system. It runs inside a MediatR pipeline with logging and FluentValidation behaviours, and follows a strict ordering: acquire dedup lock → attach domain event → persist → side effects.

Deduplication lock

Before any write, Redis SET NX on key telemetry:dedup:{packetId} with a 10-minute TTL. A duplicate packet ID returns early — counted in telemetry.duplicate_skipped metrics — without touching Postgres. 

// RedisDeduplicationService
await db.StringSetAsync(dedupKey, "1", LockExpiration, When.NotExists);
// false → duplicate, skip processing

Lock release on failure matters. If SaveChangesAsync throws, the handler deletes the dedup key so a legitimate retry can re-acquire the lock. Without this, a transient DB blip would block that packet for 10 minutes.

Side effects after commit

Cache update, anomaly analysis, and SignalR broadcast happen after successful persistence. Cache and broadcast failures are logged but do not fail the command — the authoritative record is already in Postgres.

Step Store On failure
Dedup lock Redis Redis down → proceed (fail-open) to avoid blocking ingestion
Persist + outbox PostgreSQL Release lock, throw transient → retry
Latest cache Redis Log warning, continue
Anomaly + playbook Redis window + SignalR Log error, continue
Live broadcast SignalR Log error, continue

03

Outbox — one transaction, safe retries

When telemetry is saved, a TelemetryReceivedEvent domain event is raised on the entity. An EF SaveChangesInterceptor captures all pending domain events and inserts matching OutboxMessage rows in the same transaction. A background worker polls every 2 seconds and publishes unpublished rows to fleet-telemetry-events.

Component Role
OutboxSaveChangesInterceptor Serializes domain events → OutboxMessage rows before commit
OutboxPublisherWorker Polls unpublished messages, publishes to Kafka, marks processed
OutboxMessage Id, Type, Content (JSON), OccurredOnUtc, ProcessedOnUtc

Why a fresh scope per retry

Retries used to reuse one AppDbContext across attempts. If SaveChanges failed after the interceptor added an outbox row to the change tracker, that orphaned entity stayed tracked. The retry added a second outbox row — duplicate downstream events. 

// KafkaTelemetryMessageProcessor — inside the retry loop
for (var attempt = 1; attempt <= maxAttempts; attempt++)
{
    using var scope = scopeFactory.CreateScope();  // fresh DbContext each attempt
    var mediator = scope.ServiceProvider.GetRequiredService<IMediator>();
    await mediator.Send(new ProcessTelemetryCommand(telemetry), cancellationToken);
}

This is the EF detail that matters. Interceptors mutate the change tracker before save. A failed save leaves dirty state. Scoped DbContext per attempt guarantees each retry starts with a clean tracker — exactly one outbox row on success.

04

Anomaly detection — sliding window

After persistence, each packet is pushed into a Redis list keyed by vehicle. The service keeps up to 30 points within a 30-second window and compares oldest vs newest to detect compound distress — not a single threshold, but correlated signals.

≥ 5%
Fuel drop over window
≥ 8°C
Engine temp rise + above 95°C
≥ 20 km/h
Sudden deceleration + engine > 100°C

Compound rule: fuel drop and thermal surge together → anomaly. Critical rule: hard deceleration with engine overheating → anomaly. Either triggers simulated RAG playbook orchestration over SignalR (ReceivePlaybookInstructions). 

// RedisAnomalyDetectionService
await db.ListLeftPushAsync(historyKey, JsonSerializer.Serialize(currentPoint));
await db.ListTrimAsync(historyKey, 0, MaxWindowPoints - 1);
await db.KeyExpireAsync(historyKey, WindowDuration);  // 30 seconds

Anomaly detection is best-effort after commit — a Redis outage won't roll back telemetry, it just skips analysis for that packet.

05

Real-time — SignalR hub

The hub lives at /api/stream/telemetry with camelCase JSON payloads. The server broadcasts every update to both the vehicle group and the fleet group — downstream consumers can subscribe at either granularity. 

// SignalRTelemetryBroadcastService — dual fan-out
await _hubContext.Clients.Group(dto.VehicleId)
    .SendAsync("ReceiveTelemetryUpdate", dto);
await _hubContext.Clients.Group(TelemetryHub.FleetGroupName)
    .SendAsync("ReceiveTelemetryUpdate", dto);
Direction Method / event Purpose
Client → server WatchFleet() Join fleet group — used by the Expo app
Client → server WatchVehicle(id) Join vehicle group — available for single-vehicle clients
Server → client ReceiveTelemetryUpdate Live GPS, speed, fuel, engine temp (camelCase DTO)
Server → client ReceivePlaybookInstructions Anomaly playbook response

REST endpoints (GET /api/telemetry/fleet, GET /api/telemetry/{vehicleId}) read the Redis latest-telemetry cache. History lives in Postgres; hot-path reads skip the database. When JWT_REQUIRE_AUTHENTICATION=true, hub connections require a bearer token with the vehicle:read scope.

Connection bootstrap

On mount, FleetContext loads a REST snapshot from GET /api/telemetry/fleet, then starts SignalR and invokes WatchFleet() only — joining per-vehicle groups as well would deliver duplicate events. 

// frontend/contexts/FleetContext.js
await loadFleetSnapshot();          // seed map + KPIs from Redis cache
await connection.start();
await connection.invoke('WatchFleet');
connection.on('ReceiveTelemetryUpdate', mergeTelemetry);
connection.on('ReceivePlaybookInstructions', appendAlert);

Status colours on the map are derived client-side from cached telemetry: warning when fuel < 30% or engine temp > 100°C; offline when no data yet. This is presentation logic only — anomaly detection runs server-side in Redis.

07

Following one packet end to end

To tie it together — Truck-001 sends telemetry with a sudden fuel drop and rising engine temp travelling through the whole system:

Stage What happens to that packet
Ingest Simulate API publishes JSON to fleet-telemetry with a unique packet Id.
Consume Consumer deserializes, opens processor with IServiceScopeFactory.
Dedup Redis SET NX succeeds — first time seeing this packet ID.
Persist EF saves VehicleTelemetry row; interceptor writes OutboxMessage for TelemetryReceivedEvent in same transaction.
Cache Redis key updated — fleet API and map read this on next poll.
Anomaly 30s window shows ≥5% fuel drop + ≥8°C rise → playbook orchestration fires.
Broadcast SignalR dual fan-out to vehicle + fleet groups; Expo client on WatchFleet receives update.
Outbox worker Within ~2s, event published to fleet-telemetry-events for downstream consumers.
Mobile UI Map marker moves, Fleet KPIs refresh, Alerts tab appends playbook instructions.

Reliability is layered, not single-point. Dedup prevents double-processing. Outbox prevents lost domain events. Fresh scopes prevent duplicate outbox rows on retry. DLQ captures everything that can't be salvaged. Each layer handles a different failure mode — together they keep the pipeline honest.
